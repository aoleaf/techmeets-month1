# 問い合わせフォーム（contact.php）

## 見つけた問題点

### 1. 確認画面が存在せず、即完了画面に遷移していた

最初の実装では `$submitted = true` になった時点で「送信完了」扱いの画面を表示していた。
ユーザーが入力内容を見直す機会がなく、誤送信が起きやすい設計になっていた。

### 2. 送信ボタンに二重送信防止がなかった

送信ボタンをクリックしても disabled にならないため、複数回クリックすると同じ内容が繰り返し送信される可能性があった。

### 3. 完了画面に問い合わせ内容が表示されなかった

完了画面には「受け付けました」メッセージのみで、ユーザーが何を送ったか確認できなかった。

---

## 修正した内容

### 2段階 POST による画面遷移（input → confirm → complete）

| ステップ | hidden フィールド `step` の値 | 処理内容 |
|----------|-------------------------------|----------|
| ① 入力画面 | `input` | バリデーションを実施。通過したら `confirm` へ |
| ② 確認画面 | `confirm` | 入力値を hidden で引き継いで表示。送信で `complete` へ |
| ③ 完了画面 | `complete` | 送信済みの内容を一覧表示 |

「入力内容を修正する」ボタンは別フォームで `step=input` を POST し、入力値を復元した状態でフォームに戻る。

### 送信ボタンの disabled 化

```js
form.addEventListener('submit', function () {
    btn.disabled = true;
    btn.textContent = '送信中...';
});
```

`onclick` ではなく `submit` イベントで disabled にすることで、フォームデータが送信された後に無効化される。`onclick` で先に disabled にするとブラウザによってはデータが送信されないケースがある。

### 完了画面への問い合わせ内容の表示

確認画面から POST される hidden フィールドの値（`$name` / `$email` / `$subject` / `$message`）は完了画面でも `$_POST` から取得できるため、既存の `.confirm-table` を流用してそのまま表示した。

## 学んだこと

### hidden フィールドで状態を引き継ぐ

PHP はリクエストをまたいで変数を保持しない（ステートレス）。画面間でデータを渡すには `<input type="hidden">` を使って POST に含める。セッション（`$_SESSION`）を使う方法もあるが、hidden フィールドはシンプルで実装コストが低い。

### バリデーションは「確認 → 送信」でも必要になりえる

今回は確認画面の POST（`step=confirm`）でバリデーションを省略したが、実際のサービスでは hidden フィールドの値がブラウザの開発者ツールで書き換えられる可能性がある。本番実装ではサーバー側で再バリデーションするか、セッションに値を保存してサーバー管理にするのが安全。

### XSS 対策は「出力時」に行う

`htmlspecialchars()` は入力時ではなく HTML として出力する直前にかける。入力値を加工してしまうと、メール本文やデータベースへの保存時に文字化けや二重エスケープが起きる。

### `submit` イベントと `click` イベントの違い

ボタンの disabled 制御を `onclick` で行うと、ブラウザが「このボタンは無効」と判断してフォーム送信をキャンセルするケースがある。`form.addEventListener('submit', ...)` はフォーム送信のトリガー後に処理されるため、確実に送信してからボタンを無効化できる。
