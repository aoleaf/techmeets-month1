# 問い合わせフォーム（contact.php）

## 概要

PHP 単一ファイルで実装した問い合わせフォーム。入力 → 確認 → 完了の 3 ステップ遷移を `step` という hidden フィールドで制御している。

| 機能 | 実装方法 |
|------|----------|
| 3 ステップ遷移 | hidden フィールド `step` の値で PHP 側の分岐を切り替え |
| サーバーサイドバリデーション | 必須チェック・`filter_var` によるメール形式チェック |
| XSS 対策 | `h()` ヘルパー関数（`htmlspecialchars`）を出力時に適用 |
| 入力値の保持 | エラー時・戻るボタン押下時に hidden で値を引き継ぎ |
| 二重送信防止 | `submit` イベントでボタンを disabled 化 |
| ステップ表示 | CSS で `active` / `done` を切り替えるプログレスバー |

---

## 工夫した点

### ステップインジケーターで現在位置を視覚化

```php
<div class="step-item <?= $step === 'input' ? 'active' : 'done' ?>">① 入力</div>
```

`$step` の値に応じて `active`（現在）・`done`（完了済み）・無印（未到達）の 3 状態を CSS で切り替えた。ユーザーが「今どこにいるか」を把握しやすくなる。

### インラインエラー表示とフィールドの赤枠

```php
class="<?= isset($errors['name']) ? 'is-invalid' : '' ?>"
```

エラーのあるフィールドだけ `is-invalid` クラスを付与してボーダーを赤くし、入力欄の直下にエラーメッセージを配置した。ページ上部にまとめてエラー一覧を出す方法より、どのフィールドを直せばよいか一目でわかる。

### `novalidate` でブラウザバリデーションを無効化

```html
<form method="POST" action="contact.php" novalidate>
```

`novalidate` を付けないとブラウザ独自のバリデーション UI が表示されてしまい、サーバー側のエラーメッセージと二重に出る。`novalidate` でブラウザ側を抑制し、表示を統一した。

### `white-space: pre-wrap` でメッセージの改行を保持

```css
.confirm-table td { white-space: pre-wrap; word-break: break-word; }
```

テキストエリアで入力した改行は、HTML に出力すると通常は消える。`pre-wrap` を指定することで確認画面・完了画面でも入力時の改行をそのまま表示できる。

### 修正ボタンで入力値を復元

確認画面の「← 入力内容を修正する」ボタンは、入力値を hidden で保持したまま `step=input` を POST するフォームになっている。これにより、入力画面に戻ったときに値が消えず、ユーザーが再入力する手間をなくせる。

## 見つけた問題点

### 1. 確認画面が存在せず、即完了画面に遷移していた

最初の実装では `$submitted = true` になった時点で「送信完了」扱いの画面を表示していた。
ユーザーが入力内容を見直す機会がなく、誤送信が起きやすい設計になっていた。

### 2. 送信ボタンに二重送信防止がなかった

送信ボタンをクリックしても disabled にならないため、複数回クリックすると同じ内容が繰り返し送信される可能性があった。

### 3. 完了画面に問い合わせ内容が表示されなかった

完了画面には「受け付けました」メッセージのみで、ユーザーが何を送ったか確認できなかった。

## 修正した内容

### 2段階 POST による画面遷移（input → confirm → complete）

| ステップ | hidden フィールド `step` の値 | 処理内容 |
|----------|-------------------------------|----------|
| ① 入力画面 | `input` | バリデーションを実施。通過したら `confirm` へ |
| ② 確認画面 | `confirm` | 入力値を hidden で引き継いで表示。送信で `complete` へ |
| ③ 完了画面 | `complete` | 送信済みの内容を一覧表示 |

「入力内容を修正する」ボタンは別フォームで `step=input` を POST し、入力値を復元した状態でフォームに戻る。

### 送信ボタンの disabled 化

```js
form.addEventListener('submit', function () {
    btn.disabled = true;
    btn.textContent = '送信中...';
});
```

`onclick` ではなく `submit` イベントで disabled にすることで、フォームデータが送信された後に無効化される。`onclick` で先に disabled にするとブラウザによってはデータが送信されないケースがある。

### 完了画面への問い合わせ内容の表示

確認画面から POST される hidden フィールドの値（`$name` / `$email` / `$subject` / `$message`）は完了画面でも `$_POST` から取得できるため、既存の `.confirm-table` を流用してそのまま表示した。

## 学んだこと

### hidden フィールドで状態を引き継ぐ

PHP はリクエストをまたいで変数を保持しない（ステートレス）。画面間でデータを渡すには `<input type="hidden">` を使って POST に含める。セッション（`$_SESSION`）を使う方法もあるが、hidden フィールドはシンプルで実装コストが低い。

### バリデーションは「確認 → 送信」でも必要になりえる

今回は確認画面の POST（`step=confirm`）でバリデーションを省略したが、実際のサービスでは hidden フィールドの値がブラウザの開発者ツールで書き換えられる可能性がある。本番実装ではサーバー側で再バリデーションするか、セッションに値を保存してサーバー管理にするのが安全。

### XSS 対策は「出力時」に行う

`htmlspecialchars()` は入力時ではなく HTML として出力する直前にかける。入力値を加工してしまうと、メール本文やデータベースへの保存時に文字化けや二重エスケープが起きる。

### `submit` イベントと `click` イベントの違い

ボタンの disabled 制御を `onclick` で行うと、ブラウザが「このボタンは無効」と判断してフォーム送信をキャンセルするケースがある。`form.addEventListener('submit', ...)` はフォーム送信のトリガー後に処理されるため、確実に送信してからボタンを無効化できる。

### `filter_var()` でメール形式を検証する

```php
filter_var($email, FILTER_VALIDATE_EMAIL)
```

PHP 組み込みの `filter_var` 関数に `FILTER_VALIDATE_EMAIL` を渡すと、RFC に準拠したメールアドレス形式かどうかを検証できる。自前で正規表現を書くよりも安全で読みやすい。

### `??`（null 合体演算子）で未定義時のデフォルト値を設定

```php
$name = trim($_POST['name'] ?? '');
```

`$_POST['name']` が存在しない（初回アクセス時など）場合に `??` の右辺の値が使われる。`isset()` で確認してから取得する書き方をワンライナーで書ける PHP 7 以降の演算子。

### `trim()` で入力値の前後空白を除去する

```php
$name = trim($_POST['name'] ?? '');
```

全角・半角スペースや改行だけを入力した場合に必須チェックを通り抜けないよう、`trim()` を最初にかけておく。バリデーション前に入力値を正規化する定石。

### `autocomplete` 属性でブラウザの自動補完を制御する

```html
<input type="text" name="name" autocomplete="name">
```

`autocomplete` 属性に `"name"` や `"email"` などのトークンを指定すると、ブラウザがユーザーの保存済み情報を補完候補として表示する。ユーザーが毎回入力する手間を減らせる UX の改善策。
